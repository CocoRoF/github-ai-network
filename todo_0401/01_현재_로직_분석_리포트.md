## 1. 시스템 개요

GitHub AI Network는 GitHub API를 통해 리포지토리, 개발자, 토픽 데이터를 **끊임없이 확장 수집**하며,
이들 사이의 **연결 네트워크 그래프**를 구축·시각화하는 도구이다.

**핵심 목표:** 서버에 부담을 주지 않으면서 네트워크를 무한하게 넓혀가는 것.

### 아키텍처 구성

```
┌──────────────────────────────────────────────────────────────┐
│  Frontend (React + Vite)                                     │
│  ├─ GraphPage.jsx   → 그래프 시각화 (react-force-graph-2d)   │
│  ├─ AdminPage.jsx   → 크롤링 세션 관리                       │
│  ├─ GraphView.jsx   → ForceGraph2D 렌더링 컴포넌트           │
│  ├─ Sidebar.jsx     → 필터, 노드 상세, 검색                  │
│  ├─ TableView.jsx   → 테이블 뷰 (리스트)                     │
│  └─ StatsModal.jsx  → 통계 모달                              │

│  └─ GraphCache      → TTL 기반 인메모리 캐시 (30s)           │
├──────────────────────────────────────────────────────────────┤
│  Database (SQLite + aiosqlite)                               │
│  └─ 11개 테이블: authors, repositories, topics,              │

│     crawl_tasks, crawl_logs, session_repositories,           │
│     session_authors                                          │
└──────────────────────────────────────────────────────────────┘
```

## 2. 데이터 모델 분석

### 2.1 핵심 엔티티

| 모델 | 역할 | 주요 필드 |
|------|------|-----------|
| **Author** | GitHub 사용자/조직 | github_id, login, name, avatar_url, bio, company, location, followers, following, public_repos |
| **Repository** | GitHub 리포지토리 | github_id, full_name, description, owner_id(FK→Author), stars, forks_count, language, license_name, is_fork, fork_source_id(자기참조FK) |
| **Topic** | GitHub 토픽 태그 | name (unique) |

### 2.2 관계 테이블

| 모델 | 관계 | 의미 |
|------|------|------|
| **RepoTopic** | Repository ↔ Topic (N:M) | 리포지토리에 태그된 토픽 |
| **RepoContributor** | Repository ↔ Author (N:M) + contributions | 리포지토리의 기여자와 기여 횟수 |
| **SessionRepository** | CrawlSession ↔ Repository (N:M) | 특정 세션에서 수집한 리포지토리 |
| **SessionAuthor** | CrawlSession ↔ Author (N:M) | 특정 세션에서 수집한 작성자 |

### 2.3 크롤링 제어 모델

| 모델 | 역할 |
|------|------|
| **CrawlSession** | 크롤링 세션 단위 (시드 타입, 시드 값, 상태, 각종 카운터) |
| **CrawlTask** | 개별 크롤링 작업 (session_id, task_type, target, depth, priority, status) |
| **CrawlLog** | 이벤트 기반 활동 로그 (level, event_type, message, metadata) |

## 3. 크롤링 로직 심층 분석

### 3.1 크롤링 시작 (Seed 메커니즘)

크롤링은 **CrawlSession 생성** 시 시작된다. 3가지 시드 타입을 지원한다:

| seed_type | 초기 task_type | 설명 |
|-----------|---------------|------|
| `search_query` | `search_repos` | GitHub 검색 쿼리로 리포지토리 검색 (예: `topic:machine-learning stars:>1000`) |
| `repository` | `fetch_repo` | 특정 리포지토리부터 시작 (예: `pytorch/pytorch`) |
| `user` | `fetch_user` | 특정 사용자부터 시작 (예: `karpathy`) |

세션 생성 시 초기 task 1개가 `depth=0, priority=1000`으로 생성되며, Worker가 자동 시작된다.

### 3.2 Worker 아키텍처

```
CrawlerManager
  ├─ _run()          ← 단일 asyncio Task (Worker)
  ├─ _next_task()    ← "running" 세션의 pending task를 priority DESC, created_at ASC로 선택
  ├─ _process()      ← task_type별 핸들러 디스패치
  └─ GitHubClient    ← HTTP 요청 + Rate Limit 관리
```

**핵심 특성:**
- **단일 Worker**: 전체 시스템에서 하나의 Worker만 동작. 여러 세션이 동시에 `running`이어도 하나의 Worker가 순차 처리.
- **우선순위 기반**: `priority DESC` → stars가 높은 리포지토리의 task가 먼저 처리됨.
- **중복 방지**: `_add_task()`에서 `(session_id, task_type, target)` 조합의 UniqueConstraint로 중복 task 생성 방지.
- **자동 재개**: 서버 재시작 시 `auto_resume()`이 `running` 상태 세션을 감지하고, `processing` 상태 task를 `pending`으로 리셋 후 Worker 재시작.

### 3.3 Task 처리 흐름 (4가지 핸들러)

#### 3.3.1 `search_repos` — 리포지토리 검색

```
입력: GitHub 검색 쿼리 (예: "topic:machine-learning stars:>1000")
처리:
  1. GitHub Search API로 최대 3페이지(30개/페이지, 최대 90개) 검색
  2. 각 검색 결과 리포지토리를 DB에 저장 (_save_repo)
  3. 세션-리포지토리 연결 생성
  4. 각 리포지토리에 대해 fetch_contributors task 생성
     - priority = min(stars, 10000)
     - depth = 현재 depth + 1
출력: 저장된 리포지토리 수
```

#### 3.3.2 `fetch_repo` — 단일 리포지토리 상세 조회

```
입력: 리포지토리 full_name (예: "pytorch/pytorch")
처리:
  1. GitHub API로 리포지토리 상세 정보 조회
  2. 리포지토리 DB 저장 + 세션 연결
  3. 소유자(owner) 정보 저장 + fetch_user task 생성 (priority=100)
  4. [Fork인 경우] 원본 리포지토리 저장 + fetch_repo task 생성 (priority=500)
  5. fetch_contributors task 생성 (priority = min(stars, 10000))
출력: 1 (단일 리포지토리)
```

#### 3.3.3 `fetch_user` — 사용자 상세 조회 + 리포지토리 확장

```
입력: 사용자 login (예: "karpathy")
처리:
  1. GitHub API로 사용자 상세 정보 조회 → DB 저장 (detailed=True)
  2. 사용자의 공개 리포지토리 조회 (최대 30개, stars 정렬)
  3. 필터링: AI 관련 OR stars > 50인 리포지토리만 수집
  4. 통과한 리포지토리 각각에 대해:
     - DB 저장 + 세션 연결
     - fetch_contributors task 생성 (priority = min(stars, 5000))
출력: 1 + 수집된 리포지토리 수
```

**_is_ai_related() 필터 로직:**
```
True if:
  - 리포지토리 topics에 AI_TOPICS 목록 중 하나 포함 (20개 토픽)
  - description에 AI_KEYWORDS 중 하나 포함 (10개 키워드)
  - name에 ["ml", "ai", "neural", "llm", "gpt", "bert", "transformer"] 포함
  - stars > 500  ← ⚠️ 불필요한 조건: 네트워크 확장을 인위적으로 제한함
```

> **⚠️ 현재 문제:** `_is_ai_related()`와 `stars > 50` 조건이 네트워크 확장의 병목으로 작용.
> 사용자의 리포지토리 중 이 필터를 통과하지 못한 것은 수집되지 않으며,
> 해당 리포지토리의 기여자 체인도 탐색되지 않아 네트워크 확장이 조기 중단됨.

#### 3.3.4 `fetch_contributors` — 기여자 수집

```
입력: 리포지토리 full_name
처리:
  1. GitHub API로 기여자 목록 조회 (상위 15명)
  2. 각 기여자에 대해:
     - Author DB 저장 (기본 정보만)
     - RepoContributor 관계 저장 (contributions 횟수 포함)
     - 세션-작성자 연결
     - fetch_user task 생성 (priority = min(contributions, 1000))
출력: 기여자 수
```

### 3.4 크롤링 확장 흐름 (BFS-like 탐색)

전체 크롤링은 **너비 우선 탐색(BFS)과 유사한 패턴**이지만, 우선순위 큐로 동작하여 실제로는 **Best-First Search**에 가깝다.

```
                        [Seed: search_query]
                               │
                    ┌──────────┴──────────┐
                    ▼                     ▼
            search_repos            search_repos (page 2,3)
                    │
         ┌──────────┼──────────┐
         ▼          ▼          ▼
     save_repo  save_repo  save_repo        ← depth 0
         │          │          │
         ▼          ▼          ▼
  fetch_contributors                        ← depth 1
         │
    ┌────┼────┐
    ▼    ▼    ▼
 save_author (contributor)
    │    │    │
    ▼    ▼    ▼
 fetch_user                                 ← depth 2
    │
    ├─ [AI관련 or stars>50 리포 수집]
    │    │
    │    ▼
    │  fetch_contributors                   ← depth 3
    │    │
    │    ▼
    │  fetch_user                           ← depth 4
    │    └─ ... (무한 확장 가능)
    │
    └─ [비관련 리포 스킵]
```

**핵심 관찰:**
1. **무한 확장 설계**: depth를 추적하지만 제한(max_depth)이 없어 네트워크를 끊임없이 넓혀감. 이는 프로젝트의 핵심 목표에 부합하는 설계.
2. **중복 방지가 자연적 수렴 장치**: `(session_id, task_type, target)` UniqueConstraint가 이미 방문한 엔티티의 재방문을 방지. 특정 커뮤니티/생태계는 유한한 참여자 풀을 가지므로, 중복 방지만으로도 특정 생태계 내에서는 자연스럽게 수렴함.
3. **우선순위가 탐색 방향 결정**: stars와 contributions가 높은 노드부터 처리 → 중요도 높은 노드를 먼저 발견.
4. **⚠️ stars 기반 필터가 확장을 제한**: `_is_ai_related()`의 `stars > 500` 및 `fetch_user`의 `stars > 50` 조건이 네트워크 확장 경로를 인위적으로 차단. 필터를 통과하지 못한 리포지토리 → 해당 기여자 체인 미탐색 → 네트워크 사각지대 발생.

### 3.5 데이터 저장 로직

#### `_save_repo()` (리포지토리 저장)
```
1. github_id로 기존 레코드 검색
2. 없으면 새로 생성, 있으면 stars/forks/watchers/description 갱신
3. owner 정보도 함께 저장 (Author upsert)
4. repo_created_at, repo_updated_at 파싱 및 저장
5. topics 배열 처리: Topic upsert + RepoTopic 관계 생성
6. crawled_at 타임스탬프 갱신
```

#### `_save_author()` (작성자 저장)
```
1. github_id로 기존 레코드 검색
2. 없으면 최소 정보(login, avatar_url, user_type)로 생성
3. detailed=True일 때만: name, bio, company, location, followers, following, public_repos 갱신
```

### 3.6 Rate Limit 관리

```
GitHubClient._request():
  1. 매 요청 전 crawler_delay(기본 2초) 대기
  2. 응답 헤더에서 X-RateLimit-Remaining 추적
  3. remaining < 10이면: reset 시간까지 sleep
  4. 403 Forbidden: reset 시간까지 sleep (최소 60초)
  5. 실패 시 최대 3회 재시도 (지수 백오프: 5s, 10s, 15s)
  6. 404: 무시 (None 반환)
  7. 422: 로그 후 None 반환
```

## 4. 그래프 생성 로직 심층 분석

### 4.1 GraphBuilder.build_graph() — 메인 그래프 생성

#### 입력 파라미터
| 파라미터 | 기본값 | 설명 |
|---------|-------|------|
| limit | 300 | 조회할 리포지토리 최대 수 |
| min_stars | 0 | 최소 스타 수 필터 |
| node_types | ["author","repo","topic"] | 포함할 노드 타입 |
| search | None | full_name/description 검색 (LIKE) |
| session_id | None | 특정 세션의 데이터만 필터 |
| language | None | 프로그래밍 언어 필터 |
| compact | False | 간소화 모드 (URL/description 생략) |

#### 그래프 생성 단계

```
Step 1: Repository 노드 수집
  ├─ stars DESC 정렬로 상위 N개 리포지토리 조회
  ├─ 필터 적용: min_stars, search, language, session_id
  └─ 각 리포지토리 → repo:{id} 노드 (val = sqrt(stars))

Step 2: Author (Owner) 노드 + "owns" 링크
  ├─ 수집된 리포지토리의 owner_id 집합 생성
  ├─ 해당 Author 레코드 조회
  └─ author:{id} → repo:{id} "owns" 링크 생성

Step 3: Topic 노드 + "has_topic" 링크
  ├─ RepoTopic에서 현재 리포지토리들의 토픽 조회
  ├─ topic_repo_count 집계 (토픽별 연결 리포 수)
  └─ repo:{id} → topic:{id} "has_topic" 링크 생성

Step 4: "contributes" 링크
  ├─ RepoContributor에서 현재 repo + author 범위 내 관계 조회
  ├─ "owns" 링크와 중복되지 않도록 필터링
  └─ author:{id} → repo:{id} "contributes" 링크 (weight = min(contributions/100, 3))

Step 5: "forked_from" 링크
  └─ fork_source_id가 있고 대상이 현재 노드 집합에 있으면 링크 생성

Step 6: "coworker" 링크
  ├─ SQL 집계: 같은 리포지토리에 contribute한 author 쌍 (공유 리포 ≥ 2)
  └─ author:{id} → author:{id} "coworker" 링크 (weight = min(shared_repos/3, 3))

Step 7: 노드 크기 보정
  ├─ 연결 수 집계
  └─ val += log2(1 + connections) * 0.5
```

### 4.2 생성되는 링크(Edge) 종류

| 링크 타입 | Source → Target | 의미 | 가중치 |
|----------|----------------|------|--------|
| **owns** | Author → Repo | 소유 관계 | 없음 |
| **has_topic** | Repo → Topic | 토픽 태그 관계 | 없음 |
| **contributes** | Author → Repo | 기여 관계 | min(contributions/100, 3) |
| **forked_from** | Repo → Repo | 포크 원본 관계 | 없음 |
| **coworker** | Author → Author | 동료 관계 (공유 리포 ≥ 2) | min(shared_repos/3, 3) |

### 4.3 ⚠️ Author 간 연결(coworker/contributes) 생성 실패 분석

현재 `build_graph()`의 구조적 문제로 인해 **Author 간 연결이 거의 생성되지 않는다**.
이 문제는 3단계로 연쇄적으로 발생한다:

```
━━━ 문제의 연쇄 ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

[1단계] Author 노드 = Owner만 포함
  │  Step 2에서 owner_ids만 수집
  │  → 기여자(contributor)는 DB에 있지만 그래프 노드에 없음
  │
  │  예: pytorch 리포에 contributor A, B, C가 있지만
  │      A가 다른 리포의 owner가 아니면 노드 자체가 없음
  ▼
[2단계] contributes 링크 대부분 탈락
  │  Step 4에서 `if src in node_ids` 검사
  │  → contributor Author가 node_ids에 없으므로 링크 생성 실패
  │
  │  예: author:A → repo:pytorch "contributes" 링크를 만들려하지만
  │      author:A가 node_ids에 없으므로 스킵
  ▼
[3단계] coworker 링크도 실패
  │  Step 6에서 author_db_ids = owner만 포함된 리스트
  │  → SQL이 owner들 간의 관계만 검색
  │  → 서로 다른 리포의 owner가 같은 리포에 contribute한 경우만 발견
  │  → HAVING COUNT(*) >= 2 조건 → owner 2명이 2개 이상 리포에 동시 기여해야 함
  ▼
[결과] 실제로 생성되는 coworker 링크: 거의 0개

━━━ 구체적 예시 ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

DB 상태:
  repo_contributors: A→pytorch, B→pytorch, C→pytorch (3명 contributor)
  repositories: pytorch(owner=Meta), transformers(owner=HuggingFace)

build_graph() 실행:
  Step 2: owner_ids = {Meta.id, HuggingFace.id}  ← A, B, C는 owner가 아니므로 제외
  Step 4: author_db_ids = [Meta.id, HuggingFace.id]
          WHERE author_id IN (Meta, HuggingFace) → A,B,C의 contributes 링크 모두 탈락
  Step 6: coworker SQL에 Meta, HuggingFace만 전달
          Meta와 HuggingFace가 같은 리포에 2회 이상 기여? → 거의 불가능
  결과: coworker 링크 0개, contributes 링크 0개 (본인 리포 owns만)
```

**또한 `get_neighbors()`에서도 author 타입 클릭 시 coworker 링크가 없음:**
author 노드를 클릭하면 해당 사용자의 owned/contributed repo만 표시.
같은 repo에 기여한 다른 Author를 표시하거나 coworker 링크를 생성하는 로직이 없음.

### 4.5 노드 크기(val) 산정 공식

| 노드 타입 | 기본 val | 보정 |
|----------|---------|------|
| Repository | `max(1, stars^0.5)` | + log2(1+connections) × 0.5 |
| Author | `max(1, followers^0.4)` | + log2(1+connections) × 0.5 |
| Topic | `max(1, repo_count^0.6)` | + log2(1+connections) × 0.5 |

### 4.6 get_neighbors() — 이웃 노드 탐색

특정 노드를 클릭했을 때 해당 노드의 직접 이웃을 반환한다.

| 노드 타입 | 탐색하는 이웃 | ⚠️ 누락 |
|----------|-------------|--------|
| repo | owner + topics + contributors (전부) + fork parent | ✔️ contributor를 노드로 포함 (build_graph와 다름) |
| author | 소유 리포(상위 20) + 기여 리포(상위 20) | **coworker 링크 없음**, 동료 개발자 노드 없음 |
| topic | 연결 리포(상위 50) | - |

> **⚠️ 핵심 문제:** author 타입 get_neighbors()에 coworker 탐색 로직이 완전히 없음.
> 같은 repo에 기여한 다른 Author를 보여주지 않으며, Author→Author 링크도 생성하지 않음.

### 4.7 캐시 전략

```
GraphCache:
  ├─ TTL: 30초 (기본)
  ├─ 키: "{session_id}:{limit}:{min_stars}:{types}:{search}:{language}:{compact}"
  ├─ crawler task 완료 시 전체 캐시 무효화 (invalidate_cache_fn)
  └─ 인메모리 dict 기반 (별도 외부 캐시 없음)
```

## 6. Observability (관측성) 분석

### 6.1 이벤트 로깅 시스템

CrawlerManager는 dual-write 로깅을 수행:
1. **DB 로그** (`crawl_logs` 테이블): 영구 저장, session_id별 필터 가능
2. **인메모리 버퍼** (`_log_buffer`, 최근 200개): DB 없이 빠른 조회

### 6.2 추적 가능한 이벤트

| 이벤트 | 설명 |
|--------|------|
| worker_start/stop/crash | Worker 생명주기 |
| session_start/pause/resume | 세션 상태 변경 |
| task_start/done/error | 개별 task 처리 |
| rate_limit_wait/resume | Rate Limit 대기/재개 |
| api_error | API 호출 오류 |

### 6.3 실시간 상태

`get_status()`가 반환하는 정보:
- Worker 상태 (running, healthy, uptime, heartbeat)
- 현재 처리 중인 task
- 분당 처리 속도 (5분 윈도우 기반)
- Rate Limit 상태 (remaining, limit, reset, waiting)
- 전체 API 호출 수
- 마지막 에러

## 9. 결론

현재 시스템은 **세션 기반의 BFS/Best-First 하이브리드 크롤링 엔진**으로,
GitHub의 리포지토리-개발자-토픽 네트워크를 무한히 탐색하며 6가지 관계 유형의 그래프를 구축한다.
단일 Worker 구조로 Rate Limit을 안전하게 관리하며, 우선순위 기반 task 큐로 중요한 노드부터 탐색한다.
이벤트 로깅과 실시간 상태 모니터링으로 관측성도 확보하고 있다.

**핵심 강점:** 중복 방지 기반의 자연스러운 수렴 + 서버 부담을 주지 않는 Rate Limit 관리로 무한 확장 가능.

**주요 병목 2가지:**
1. **stars 필터가 확장 차단**: `_is_ai_related()`와 `stars > 50` 필터가 네트워크 확장 경로를 인위적으로 차단하여 사각지대 발생.
2. **Author 간 연결이 사실상 작동하지 않음**: `build_graph()`가 **owner만 Author 노드로 포함**하기 때문에, contributes 링크와 coworker 링크가 거의 생성되지 않음. 이는 네트워크 그래프의 핵심 가치(개발자 간 관계 시각화)를 훼손하는 구조적 결함.
