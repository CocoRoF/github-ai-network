# 3D 그래프 렌더링 최적화 전략

## 1. 현재 상태 진단

### 1.1 증상
- Max Nodes 100만 설정 시 실제 약 30,000개 노드에서 **페이지 완전 프리즈**
- 브라우저 메인 스레드가 블로킹되어 탭 자체가 응답 불가

### 1.2 현재 아키텍처 병목 분석

| 병목 | 위치 | 심각도 | 설명 |
|------|------|--------|------|
| **nodeThreeObject 개별 생성** | GraphView3D.jsx | ★★★★★ | 30K 노드 × (Group + Mesh + SpriteText) = **90K+ Three.js 오브젝트**. 각각 별도 draw call 발생 |
| **Force 시뮬레이션 메인스레드** | react-force-graph-3d 내부 | ★★★★★ | d3-force-3d가 30K 노드의 물리연산을 **메인 스레드에서 동기 실행**. warmupTicks 동안 완전 블로킹 |
| **SpriteText 텍스처 폭발** | nodeThreeObject 내부 | ★★★★☆ | 노드마다 Canvas → Texture 생성. 30K개 = **30K개 GPU 텍스처** 업로드 |
| **Bloom 포스트프로세싱** | UnrealBloomPass | ★★★☆☆ | 전체 화면 blur 3패스. 오브젝트 수와 무관하지만 GPU fillrate 소모 |
| **Raycasting (포인터 인터랙션)** | enablePointerInteraction | ★★★☆☆ | 매 프레임 30K 오브젝트 대상 ray intersection 검사 |
| **Star Field + Nebula** | createStarField / createNebula | ★★☆☆☆ | 10K 파티클 + 30 스프라이트. 상대적으로 가벼움 |
| **Adjacency 리빌드** | highlightSet useMemo | ★★☆☆☆ | 노드 선택 시 30K links 순회. O(n) 수준으로 심각하지 않음 |

### 1.3 Draw Call 계산 (현재)

```
30,000 노드 × nodeThreeObject:
  - Group: 1 (컨테이너, draw call 없음)
  - Mesh (SphereGeometry): 1 draw call
  - SpriteText (라벨): 1 draw call (nodeCount < 50K이면 표시)
  ≈ 60,000 draw calls

+ 링크 (약 50,000개 추정): 각 Line/Cylinder = 1 draw call
  ≈ 50,000 draw calls

+ Star field: 1 draw call (Points)
+ Nebula: 30 draw calls
+ Bloom: 3 fullscreen passes

총합 ≈ 110,000+ draw calls/frame
```

> **WebGL 권장 draw call 수: 500~2,000개**. 현재 구현은 50배 이상 초과.

## 2. 최적화 전략 (우선순위 순)

### Phase 1: InstancedMesh 전환 (Draw Call 99% 감소) ★★★★★

**핵심 원리**: 같은 geometry + material을 공유하는 오브젝트를 GPU가 한 번의 draw call로 일괄 렌더링

**현재**:
```
30,000개 노드 → 30,000 Mesh → 30,000 draw calls
```

**개선 후**:
```
3개 InstancedMesh (author/repo/topic) → 3 draw calls
```

#### 구현 방안

```javascript
// nodeThreeObject 대신 커스텀 렌더링 레이어 사용
const instancedMeshes = useMemo(() => {
  const types = ['author', 'repo', 'topic'];
  const meshes = {};

  types.forEach(type => {
    const nodes = graphData.nodes.filter(n => n.type === type);
    const geo = new THREE.SphereGeometry(1, 6, 4);  // 공유 지오메트리
    const mat = new THREE.MeshStandardMaterial({
      color: NODE_COLORS[type],
      emissive: NODE_COLORS[type],
      emissiveIntensity: 0.35,
    });
    const mesh = new THREE.InstancedMesh(geo, mat, nodes.length);
    mesh.instanceMatrix.setUsage(THREE.DynamicDrawUsage);  // 위치 업데이트 최적화
    meshes[type] = { mesh, nodes };
  });

  return meshes;
}, [graphData.nodes]);

// 매 프레임 위치만 업데이트 (nodePositionUpdate 활용)
function updateInstancePositions() {
  const dummy = new THREE.Object3D();
  Object.values(instancedMeshes).forEach(({ mesh, nodes }) => {
    nodes.forEach((node, i) => {
      dummy.position.set(node.x, node.y, node.z);
      dummy.scale.setScalar(getNodeSize(node));
      dummy.updateMatrix();
      mesh.setMatrixAt(i, dummy.matrix);
    });
    mesh.instanceMatrix.needsUpdate = true;
  });
}
```

**기대 효과**:
- 노드 Draw call: 30,000 → **3**
- GPU 메모리: ~90% 감소
- 렌더링 시간: ~95% 감소

**제약사항**:
- react-force-graph-3d의 `nodeThreeObject`를 포기해야 함
- 대안 A: `nodeThreeObject` 대신 `scene()` 직접 접근해서 InstancedMesh 수동 관리
- 대안 B: react-force-graph-3d를 버리고 3d-force-graph 직접 사용 (React 래퍼 없이)
- 대안 C: nodeThreeObject는 null 반환, 별도 InstancedMesh를 scene에 추가하고 `nodePositionUpdate`로 동기화

### Phase 2: Force 시뮬레이션 Web Worker 오프로드 ★★★★★

**핵심 문제**: d3-force-3d는 메인 스레드에서 실행됨. 30K 노드 × N tick = 수 초간 UI 완전 블로킹

#### 방안 A: 서버 사이드 레이아웃 사전계산 (권장)

```python
# backend에서 igraph / networkx로 레이아웃 계산
import igraph as ig

def compute_layout(nodes, links):
    g = ig.Graph()
    g.add_vertices(len(nodes))
    g.add_edges([(l['source_idx'], l['target_idx']) for l in links])

    # Fruchterman-Reingold 3D 레이아웃 (C로 구현되어 파이썬보다 100배 빠름)
    layout = g.layout_fruchterman_reingold_3d()

    for i, node in enumerate(nodes):
        node['x'], node['y'], node['z'] = layout[i]

    return nodes
```

**장점**:
- 프론트엔드 부하 **제로** (이미 위치가 계산된 상태로 전송)
- igraph C 백엔드 = Python보다 100배, JS보다 50배 빠름
- 동일 쿼리 결과 캐싱 가능 (레이아웃까지 캐싱)

**단점**:
- 인터랙티브 드래그 시 재계산 불가 (하이브리드 필요)
- 서버 메모리/CPU 사용 증가

#### 방안 B: Web Worker 기반 Force 시뮬레이션

```javascript
// force-worker.js
importScripts('https://cdn.jsdelivr.net/npm/d3-force-3d');

self.onmessage = function(e) {
  const { nodes, links, config } = e.data;

  const simulation = d3.forceSimulation(nodes)
    .numDimensions(3)
    .force('link', d3.forceLink(links).id(d => d.id))
    .force('charge', d3.forceManyBody().strength(-30))
    .force('center', d3.forceCenter())
    .stop();

  // 배치 단위로 tick 실행 + 프로그레스 리포트
  const totalTicks = 300;
  const batchSize = 10;

  for (let i = 0; i < totalTicks; i += batchSize) {
    for (let j = 0; j < batchSize; j++) simulation.tick();

    // 진행률을 메인 스레드에 보고
    self.postMessage({
      type: 'progress',
      progress: (i + batchSize) / totalTicks,
      nodes: nodes.map(n => ({ id: n.id, x: n.x, y: n.y, z: n.z }))
    });
  }

  self.postMessage({ type: 'done', nodes, links });
};
```

**장점**: 메인 스레드 완전 논블로킹, 프로그레스 표시 가능
**단점**: Worker-메인 간 데이터 직렬화 오버헤드 (Transferable 사용하면 완화)

#### 방안 C: warmupTicks ← 전체 tick, cooldownTicks ← 0

```javascript
// 렌더링 시작 전에 레이아웃 완전 계산 (프리즈는 하지만 짧게)
warmupTicks={300}
cooldownTicks={0}  // 렌더링 시에는 추가 tick 없음
```

**장점**: 구현 변경 최소. 렌더링 시작 후에는 매우 빠름
**단점**: warmup 동안 여전히 프리즈 (3~5초)

### Phase 3: 라벨 최적화 (GPU 텍스처 90% 감소) ★★★★☆

#### 문제
- SpriteText: 노드마다 Canvas.getContext('2d') → drawText → toTexture
- 30K 텍스처 = ~300MB+ GPU VRAM 소모

#### 해결: 거리 기반 동적 라벨링

```javascript
// 라벨을 항상 전부 만들지 않고, 카메라 거리 + 중요도로 필터링
const LABEL_BUDGET = 100;  // 동시에 표시할 최대 라벨 수

function getVisibleLabels(nodes, camera, budget) {
  return nodes
    .map(n => ({
      node: n,
      dist: camera.position.distanceTo(new THREE.Vector3(n.x, n.y, n.z)),
      importance: n.val || 1
    }))
    .sort((a, b) => (a.dist / a.importance) - (b.dist / b.importance))
    .slice(0, budget);
}
```

#### 대안: CSS2DRenderer 오버레이

```javascript
// Three.js 3D 라벨 대신 HTML/CSS로 2D 오버레이
// GPU 텍스처 사용 안 함, DOM 기반이라 재활용 가능
import { CSS2DRenderer, CSS2DObject } from 'three/examples/jsm/renderers/CSS2DRenderer';
```

**기대 효과**: GPU VRAM 90% 절감, 라벨 생성 시간 95% 감소

### Phase 5: Progressive Loading (점진적 렌더링) ★★★☆☆

#### 원리
한 번에 30K 노드를 넘기지 않고, 중요도 기반으로 단계적 로딩

```
Step 1: Top 500 노드 (stars 상위) → 즉시 렌더링 (< 1초)
Step 2: 1000 → 3000 → 10000 → 30000 (배치 추가)
```

#### 프론트엔드 구현

```javascript
const [visibleCount, setVisibleCount] = useState(500);

// 단계적으로 노드 수 증가
useEffect(() => {
  if (visibleCount >= graphData.nodes.length) return;

  const timer = setTimeout(() => {
    setVisibleCount(prev => Math.min(prev * 3, graphData.nodes.length));
  }, 2000);  // 2초마다 3배씩 증가

  return () => clearTimeout(timer);
}, [visibleCount, graphData.nodes.length]);

// 노드는 이미 val 기준으로 정렬되어 있음 (backend에서 stars DESC)
const visibleData = useMemo(() => ({
  nodes: graphData.nodes.slice(0, visibleCount),
  links: graphData.links.filter(l => {
    const s = l.source?.id ?? l.source;
    const t = l.target?.id ?? l.target;
    return visibleNodeIds.has(s) && visibleNodeIds.has(t);
  })
}), [graphData, visibleCount]);
```

## 5. 핵심 인사이트

### 왜 30K에서 프리즈되는가?

```
1. graphData 전달 즉시 → react-force-graph-3d가 nodeThreeObject를 30K번 호출
2. 각 호출마다 new THREE.Group() + new THREE.Mesh() + new SpriteText()
   → JS 객체 할당 + GPU 버퍼 생성 = 수 초 동안 메인 스레드 블로킹
3. 동시에 d3-force-3d가 warmupTicks만큼 물리 시뮬레이션 실행
   → 추가 수 초 블로킹
4. 첫 프레임 도달까지 10초+ 소요
5. 렌더링 시작 후에도 110K draw calls → GPU 포화 → <5fps
6. 브라우저가 "응답없음" 판정
```

### 가장 효과적인 단일 변경

**`nodeThreeObject`를 사용하지 않는 것.**

react-force-graph-3d는 `nodeThreeObject`가 없으면 내부적으로 **InstancedMesh**를 사용하여 모든 노드를 단일 draw call로 렌더링한다. 현재 코드에서 `LOD_POINT = 50000`으로 설정되어 50K 미만에서는 항상 `nodeThreeObject`를 사용하고 있어, 30K 노드에서 최악의 성능 경로를 타고 있다.

```javascript
// 변경 전
const LOD_POINT = 50000;

// 변경 후: 5000 이상이면 내장 InstancedMesh 사용
const LOD_POINT = 5000;
```

**이 한 줄 변경만으로 30K 노드 렌더링이 가능해질 가능성이 높다.**

## 6. 참고: react-force-graph-3d 내부 렌더링 전략

| nodeThreeObject 사용 | 렌더링 방식 | Draw Calls |
|---------------------|------------|------------|
