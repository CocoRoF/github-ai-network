# 3D 렌더링 2차 개선 계획

## 1. 현재 잔존 문제 3가지

### 문제 A: 여전히 심각한 렉
### 문제 B: 화면 이동 중 급격한 zoom-out
### 문제 C: 노드 선택 해제 불가 (빈 공간 클릭 시)

---

## 2. 원인 분석: 전체 파이프라인 추적

### 2.1 데이터 흐름 (렌더링 파이프라인)

```
[GraphPage] fetchGraph()
  → setGraphData(data)                    ← 30K 노드 원본
  → progressive loading effect 발동
    → setProgressiveCount(500)            ← 500개로 시작
    → setTimeout × 2.5배씩 증가           ← 500 → 1250 → 3125 → 7812 → 19531 → 30000
  → visibleData = useMemo(slice)
    → <GraphView3D graphData={visibleData}>
      → ForceGraph3D에 graphData 전달
```

### 2.2 문제 A 근본 원인: Progressive Loading이 re-mount를 반복 유발

**핵심 발견**: `visibleData`가 매 progressive step마다 **새로운 object reference**를 생성한다.

```javascript
// 매번 새 객체 → ForceGraph3D가 graphData 변경으로 인식
return { nodes, links, stats: graphData.stats };
```

react-force-graph-3d는 `graphData` prop이 변경되면:
1. **기존 force simulation을 재시작** (모든 노드 위치 리셋)
2. `nodeThreeObject`가 있으면 **모든 노드의 3D 오브젝트를 재생성**
3. `warmupTicks`만큼 다시 blocking simulation 실행
4. `onEngineStop`에서 `zoomToFit` 호출

**따라서 progressive loading의 5~6단계 × 매번 전체 재초기화 = 치명적 렉**

```
단계별 발생하는 일:
Step 1: 500 nodes → warmup 80 ticks (blocking) → custom objects 500개 생성 → render
Step 2: 1250 nodes → warmup 80 ticks (blocking) → custom objects 1250개 재생성 → render
Step 3: 3125 nodes → warmup 80 ticks (blocking) → custom objects 3125개 재생성 → render
Step 4: 7812 nodes → TIER_LARGE 초과 → ngraph 전환 → warmup 100 ticks → render
Step 5: 19531 nodes → warmup 100 ticks → render
Step 6: 30000 nodes → warmup 100 ticks → render

= 약 6번의 full re-initialization + 총 540 ticks의 blocking simulation
```

### 2.3 문제 B 근본 원인: `onEngineStop` → `zoomToFit`

```javascript
onEngineStop={() => {
  if (graphRef?.current && nodeCount > 0) {
    graphRef.current.zoomToFit(800, 100);  // ★ 매 progressive step마다 호출!
  }
}}
```

- Progressive loading의 각 step이 graphData를 변경하면 force engine이 재시작 → 멈춤 → `onEngineStop` → `zoomToFit`
- 사용자가 화면을 zoom-in해서 탐색 중인데, 갑자기 `zoomToFit`이 호출되어 **전체 그래프가 보이도록 강제 zoom-out**
- 사용자 입장에서는 "갑자기 확 빠지는" 현상으로 느껴짐

### 2.4 문제 C 근본 원인: `onBackgroundClick` vs `enablePointerInteraction`

```javascript
// GraphView3D.jsx
onBackgroundClick={() => onNodeClick(null)}  // 빈 공간 클릭 → null 전달
```

```javascript
// GraphPage.jsx
const handleNodeClick = async (node) => {
  if (!node) {
    setSelectedNode(null);  // ★ 이 경로가 실행되어야 함
    return;
  }
  ...
};
```

이 로직 자체는 맞지만, 문제는 **두 가지 상충하는 행동**:

1. **react-force-graph-3d의 `onBackgroundClick`**: TrackballControls의 마우스 이벤트와 충돌.
   - 사용자가 드래그(회전/이동)한 후 마우스를 놓으면, 라이브러리가 이것을 "click"으로 판단하지 않음
   - 반대로 정확히 같은 위치에서 mousedown → mouseup 해야 "click"으로 인식
   - 하지만 3D 공간에서는 약간의 마우스 이동이 항상 발생해서, 대부분의 "빈 공간 클릭"이 **드래그로 무시**됨

2. **ForceGraph3D 내부의 이벤트 우선순위**: 노드 위에서 클릭하면 `onNodeClick` 호출, 빈 공간이면 `onBackgroundClick` 호출인데, **raycasting이 느릴 때 (30K 노드) 이벤트 자체가 누락**될 수 있음

## 3. 개선 계획

### 3.1 Progressive Loading 완전 재설계

**현재 방식 (문제)**:
- graphData를 잘라서 전달 → ForceGraph3D가 매번 full re-init

**새로운 방식**:
- Progressive loading을 제거하고, **한 번만 전체 데이터를 전달**
- 대신 InstancedMesh 경로 (built-in)를 적극 활용하여 초기 렌더 cost를 최소화
- `warmupTicks`로 레이아웃을 사전 계산하되, 값을 적절히 제한하여 blocking 시간 최소화

```javascript
// AS-IS: progressive → 6번 re-init
// TO-BE: 1번만 전달, warmup으로 한번만 block
<ForceGraph3D graphData={graphData} warmupTicks={적절값} cooldownTicks={0} />
```

### 3.2 `onEngineStop` zoomToFit 제한

```javascript
// 최초 1회만 zoomToFit 실행, 이후는 무시
const hasZoomedRef = useRef(false);

onEngineStop={() => {
  if (!hasZoomedRef.current && graphRef?.current && nodeCount > 0) {
    graphRef.current.zoomToFit(800, 100);
    hasZoomedRef.current = true;
  }
}}
```

graphData가 실제로 변경될 때만 (사용자가 Refresh 클릭 시) 리셋:
```javascript
useEffect(() => {
  hasZoomedRef.current = false;
}, [graphData]);
```

### 3.3 노드 선택 해제: 이중 안전장치

react-force-graph-3d의 `onBackgroundClick`이 TrackballControls와 충돌하는 문제를 우회:

```javascript
// 방법 1: onBackgroundClick 유지 (작동할 때 작동)
onBackgroundClick={() => onNodeClick(null)}

// 방법 2: 키보드 Escape키로 해제 (확실한 대안)
useEffect(() => {
  const handler = (e) => {
    if (e.key === "Escape") onNodeClick(null);
  };
  window.addEventListener("keydown", handler);
  return () => window.removeEventListener("keydown", handler);
}, [onNodeClick]);

// 방법 3: GraphPage의 graph-container에 직접 click handler 추가
// ForceGraph3D의 canvas가 이벤트를 소비하지 않은 경우에만 발동
```

### 3.4 Custom nodeThreeObject 임계값을 더 낮추기

현재 `TIER_LARGE = 5000`이지만, progressive loading이 단계별로 500 → 1250 → 3125까지는 custom objects를 사용하다가 5000 넘어가면 갑자기 InstancedMesh로 전환 → **시각적 불연속 + 전환 시 재생성 비용**

```javascript
// 5K → 2K로 낮춤: 2000개 미만에서만 custom objects
const TIER_LARGE = 2000;
```

### 3.5 nodeThreeObject의 selectedNode 의존성 제거

```javascript
// 현재: selectedNode 변경 → nodeThreeObject 재생성 → 모든 노드 3D 오브젝트 재생성!
const nodeThreeObject = useCallback(
  (node) => { ... },
  [getNodeSize, selectedNode, highlightSet, nodeCount, ...]  // ★ selectedNode 포함!
);
```

`selectedNode`가 바뀔 때마다 **모든 노드의 3D 오브젝트가 처음부터 다시 생성**된다.
이것이 선택/해제 시 렉의 원인.

**해결**: selectedNode를 ref로 관리하여 nodeThreeObject의 deps에서 제거하고,
대신 노드 material의 emissive/opacity를 **직접 업데이트** (오브젝트 재생성 없이):

```javascript
const selectedNodeRef = useRef(null);
selectedNodeRef.current = selectedNode;

// nodeThreeObject에서 selectedNodeRef.current 사용
// deps에서 selectedNode, highlightSet 제거
// → 노드 선택 시 오브젝트 재생성 안 함
```

별도의 useEffect에서 기존 오브젝트의 material만 갱신:
```javascript
useEffect(() => {
  if (!graphRef?.current) return;
  // graphRef.current.nodeThreeObject()로 각 노드의 Object3D에 접근
  // mesh.material.emissiveIntensity / opacity만 변경
  graphRef.current.refresh();  // 내부 3D 오브젝트 재사용, material만 갱신
}, [selectedNode]);
```

### 3.6 Force 엔진 설정 최적화

```javascript
// 현재: d3 → 5K에서 ngraph 전환
// 문제: d3의 기본 설정이 너무 오래 돌아감

// 개선:
warmupTicks = Math.min(nodeCount * 0.5, 200);  // 노드 수 비례, 최대 200
cooldownTicks = 0;                              // 렌더 중 추가 tick 없음 (CPU 해방)
cooldownTime = 3000;                            // 최대 3초만 추가 시뮬레이션
d3AlphaDecay = 0.05;                           // 빠르게 수렴
d3VelocityDecay = 0.5;                         // 빠르게 안정화
```

### 3.7 정리: 불필요한 re-render 차단

GraphView3D에 현재 inline 함수들이 prop으로 전달되고 있음:
```javascript
// 매 render마다 새 함수 → ForceGraph3D 불필요한 re-render
onBackgroundClick={() => onNodeClick(null)}  // inline arrow
nodeLabel={(n) => n.label}                    // inline arrow
```

이들을 useCallback으로 감싸거나 stable reference로 교체.

## 4. 구현 우선순위

| 순서 | 작업 | 해결하는 문제 | 영향도 |
|------|------|--------------|--------|
| 1 | Progressive Loading 제거 → 한번에 전달 | A (렉), B (zoom-out) | ★★★★★ |
| 2 | `onEngineStop` zoomToFit 최초 1회 제한 | B (zoom-out) | ★★★★★ |
| 3 | 노드 선택 해제: Escape키 + onBackgroundClick 보강 | C (선택 해제) | ★★★★☆ |
| 4 | nodeThreeObject에서 selectedNode deps 제거 + ref 패턴 | A (선택 시 렉) | ★★★★☆ |
| 5 | TIER_LARGE 2000으로 하향 | A (렉) | ★★★☆☆ |
| 6 | inline 함수 stable reference 처리 | A (렉) | ★★☆☆☆ |
| 7 | Force 엔진 설정 미세 조정 | A (렉) | ★★☆☆☆ |

## 5. 요약

**가장 큰 단일 원인**: Progressive Loading이 graphData를 6번 교체하면서 ForceGraph3D가 매번 full re-initialization을 수행. 이것 하나를 제거하는 것만으로 렉의 80%+ 해소 예상.

**Zoom-out 원인**: `onEngineStop` → `zoomToFit`이 progressive step마다 + engine reheat마다 반복 호출.

**선택 해제 원인**: 3D TrackballControls에서 미세 마우스 이동이 click을 drag로 전환시킴 → `onBackgroundClick` 미발동.
