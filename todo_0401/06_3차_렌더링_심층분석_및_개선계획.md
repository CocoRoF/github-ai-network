## 1. 현재 아키텍처 개요

### 1.1 컴포넌트 계층

```
<React.StrictMode>            ← ★ 개발 환경에서 이중 마운트 유발
  <BrowserRouter>
    <GraphPage>               ← 데이터 fetch, 상태 관리, focusNode 카메라 이동
      <Sidebar>               ← 필터/스타일 제어, 선택 노드 정보 표시
      <Suspense>
        <GraphView3D>         ← ForceGraph3D 래핑, 씬 초기화, 3D 객체 생성
          <ForceGraph3D>      ← react-kapsule 래퍼 (React reconciliation 미사용)
            ↓ 내부 구조
            three-forcegraph  ← Three.js 씬 직접 조작 (ThreeDigest 패턴)
            3d-force-graph    ← 카메라, 렌더러, 포스트프로세싱, 이벤트
```

### 1.2 라이브러리 내부 동작 방식 (핵심)

react-force-graph-3d는 **React Virtual DOM을 사용하지 않는다**.

```
React 컴포넌트 → <div> 하나만 렌더링 (react-kapsule)
                  ↓
              kapsule 인스턴스가 Three.js Canvas를 직접 생성/관리
                  ↓
              requestAnimationFrame 루프로 렌더링 (React 외부)
                  ↓
              prop 변경 시 → kapsule 내부 메서드 직접 호출 (setState 아님)
```

**결론**: React re-render는 cheap하지만, prop 변경은 직접적으로 kapsule 메서드를 트리거한다.
따라서 **prop identity 변경 = 실제 내부 업데이트**이며, 불필요한 prop 변경이 곧 성능 저하.

### 1.3 핵심 라이브러리 메서드 동작

| 메서드 | 실제 동작 | 비용 |
|--------|----------|------|
| `refresh()` | `_flushObjects = true` → 모든 노드/링크의 Three.js 객체 재생성 | **매우 높음** (N × Three.Group + Mesh + Material) |
| `graphData` prop 변경 | 시뮬레이션 중지 → alpha(1) 재가열 → warmupTicks 실행 → 모든 객체 재생성 → 쿨다운 시작 | **극도로 높음** |
| `zoomToFit(ms, padding)` | getGraphBbox() → fitToBbox() 카메라 애니메이션 | 낮음 |
| `cameraPosition(pos, lookAt, ms)` | 카메라 위치/방향 트윈 애니메이션 | 낮음 |

## 3. 치명적 버그 (Critical)

### 3.1 BUG-C1: Division by Zero → 카메라 무한대 이동 (블랭크 스크린)

**위치**: `GraphView3D.jsx:472`, `GraphPage.jsx:104`

**코드**:
```javascript
// GraphView3D.jsx line 472
const distRatio = 1 + distance / Math.hypot(node.x || 0, node.y || 0, node.z || 0);
//                                           ↑ 모두 0이면 hypot = 0 → 120/0 = Infinity

// GraphPage.jsx line 104 (동일)
const distRatio = 1 + distance / Math.hypot(node.x || 0, node.y || 0, node.z || 0);
```

**발생 조건**:
1. `warmupTicks = 0` (nodeCount < 2000) → 노드가 origin(0,0,0)에서 시작 → 시뮬레이션 시작 직후 클릭
2. `handleExpand`로 추가된 노드 → `x`, `y`, `z`가 `undefined` → `|| 0`으로 모두 0
3. 우연히 origin에 배치된 노드 클릭

**결과**:
```
distRatio = 1 + 120 / 0 = Infinity
cameraPosition({ x: Infinity, y: Infinity, z: Infinity }, ...)
→ Three.js 카메라가 무한대 좌표로 이동
→ 모든 오브젝트가 클리핑 범위 밖 → 화면 완전 블랙
→ 사용자는 아무것도 보이지 않음, 복구 불가 (새로고침 필요)
```

**수정**:
```javascript
// Math.hypot가 0 또는 매우 작으면 안전한 기본 거리 사용
const nodePos = Math.hypot(node.x || 0, node.y || 0, node.z || 0);
if (nodePos < 1) return; // 아직 배치되지 않은 노드 → 카메라 이동 skip
const distRatio = 1 + distance / nodePos;
```

### 3.2 BUG-C2: 이중 cameraPosition 호출 (카메라 불안정)

**위치**: `GraphView3D.jsx:464-485` + `GraphPage.jsx:100-113`

**흐름**:
```
사용자 노드 클릭
  → GraphView3D.handleNodeClick (line 464)
      → graphRef.current.cameraPosition(pos1, lookAt1, 1200)  ← [1차 이동]
      → onNodeClick(node)
          → GraphPage.handleNodeClick (line 115)
              → focusNode(node) (line 100)
                  → graphRef.current.cameraPosition(pos2, lookAt2, 1200)  ← [2차 이동]
```

두 호출은 동일한 파라미터지만, 내부적으로 두 개의 독립적인 트윈 애니메이션이 시작된다.
두 번째 호출이 첫 번째를 중단(interrupt)하면서 카메라가 불안정하게 떨린다.

**수정**: GraphView3D에서 카메라 이동을 담당하고, GraphPage.focusNode는 제거하거나
GraphView3D.handleNodeClick에서 카메라 이동 제거하고 GraphPage에 일임.
핵심은 **한 곳에서만 cameraPosition을 호출**해야 한다.

### 3.3 BUG-C3: React.StrictMode + setTimeout → 씬 초기화 경합

**위치**: `main.jsx:4` (StrictMode), `GraphView3D.jsx:564-569` (ref 콜백)

**React.StrictMode 개발 모드 동작**:
```
[1단계] Mount     → ref(el_A)   → setTimeout(handleEngineInit(el_A), 100)
[2단계] Unmount   → ref(null)
[3단계] Re-mount  → ref(el_B)   → setTimeout(handleEngineInit(el_B), 100)
                                    ↑ el_B는 실제 사용되는 인스턴스

[100ms 후]
setTimeout #1 실행: handleEngineInit(el_A)
  → el_A는 이미 언마운트된 인스턴스
  → initDoneRef.current = true  ← ★ 여기서 true로 설정됨!
  → el_A.scene()에 bloom, stars 추가 (쓸모없는 인스턴스)

setTimeout #2 실행: handleEngineInit(el_B)
  → initDoneRef.current가 이미 true
  → return;  ← ★ 실제 인스턴스 el_B의 초기화가 SKIP됨!
```

**결과**: 개발 환경에서 bloom, star field, fog, lighting이 적용되지 않을 수 있음.

**수정**:
```javascript
// ref 콜백에서 직접 초기화 (setTimeout 제거)
ref={(el) => {
  if (graphRef) graphRef.current = el;
  if (el) handleEngineInit(el);  // 동기 실행
}}

// 또는 useEffect로 이동
useEffect(() => {
  if (graphRef?.current && !initDoneRef.current) {
    handleEngineInit(graphRef.current);
  }
}, [handleEngineInit, graphRef]);
```

## 4. 주요 결함 (Major)

### 4.1 BUG-M1: `refresh()`가 전체 3D 객체를 재생성

**위치**: `GraphView3D.jsx:189-193`

**2차 계획의 의도 (3.5절)**: selectedNode 변경 시 기존 mesh의 material만 갱신
**실제 구현**: `graphRef.current.refresh()` 호출

**라이브러리 내부 동작** (`three-forcegraph.mjs:692-696`):
```javascript
refresh: function(state) {
  state._flushObjects = true;  // ← 모든 객체 삭제 후 재생성 플래그
  state._rerender();
  return this;
}
```

`_flushObjects = true`이면 ThreeDigest가 nodeDataMapper를 초기화하고,
`nodeThreeObject(node)`를 **모든 노드에 대해 다시 호출**한다.

**비용 (1000 노드 기준)**:
```
× 1000: new THREE.Group()
× 1000: new THREE.MeshStandardMaterial({ ... })  ← 가장 무거움
× 1000: new THREE.Mesh(geo, mat)
×  ~80: new SpriteText(label)  ← 내부에서 Canvas + CanvasTexture 생성
= 매 선택/해제마다 ~3000개 Three.js 객체 생성 + ~3000개 이전 객체 GC
```

**수정 방안**: `refresh()` 대신 각 노드의 기존 `__threeObj`에 직접 접근하여 material만 변경:
```javascript
useEffect(() => {
  if (!graphRef?.current) return;
  const fg = graphRef.current;

  graphData.nodes.forEach((node) => {
    const obj = node.__threeObj;  // ForceGraph가 바인딩한 Three.js 객체
    if (!obj) return;

    const mesh = obj.children.find((c) => c.isMesh);
    if (!mesh) return;

    const isSelected = selectedNode?.id === node.id;
    const hopDist = highlightSet?.get(node.id);
    const isInHighlight = hopDist !== undefined;
    const dimmed = highlightSet && !isInHighlight;

    // material 속성만 변경 (객체 재생성 없음)
    mesh.material.emissiveIntensity = dimmed ? 0.02 : isSelected ? 1.2 : isInHighlight ? 0.6 : 0.35;
    mesh.material.opacity = dimmed ? 0.06 : isInHighlight ? [1, 0.9, 0.6, 0.35][hopDist] ?? 0.35 : 0.85;
    mesh.material.needsUpdate = true;

    // 선택 링 추가/제거
    const existingRing = obj.children.find((c) => c.userData?.isSelectionRing);
    if (isSelected && !existingRing) {
      // 링 추가
    } else if (!isSelected && existingRing) {
      obj.remove(existingRing);
      existingRing.geometry.dispose();
      existingRing.material.dispose();
    }
  });
}, [selectedNode, highlightSet]);
```

### 4.2 BUG-M2: `TIER_MED === TIER_LARGE = 2000` — 중간 티어 소멸

**위치**: `GraphView3D.jsx:24-25`

```javascript
const TIER_MED    = 2000;   // custom objects, selective labels, bloom
const TIER_LARGE  = 2000;   // ★ switch to built-in InstancedMesh
```

**문제**: 두 상수가 동일하여 중간 성능 티어가 존재하지 않음.

**영향 분석**:
```
nodeCount     | useCustomObj | useCurvedLinks | warmupTicks | 주석
< 500         | true         | true           | 0           | TIER_SMALL
500 ~ 1999    | true         | true           | 0           | TIER_MED == TIER_LARGE
2000 ~ 14999  | false        | false          | 100 (ngraph)| 갑자기 InstancedMesh + ngraph로 전환
15000+        | false        | false          | 100         | TIER_HUGE
```

nodeCount가 2000을 넘는 순간 **세 가지가 동시에 전환**됨:
- custom objects → InstancedMesh
- d3 force → ngraph
- curved links → straight links

이 급격한 전환은 UX 불연속을 야기한다.

**수정 권장**:
```javascript
const TIER_SMALL  = 500;    // full effects
const TIER_MED    = 2000;   // custom objects, selective labels, reduced effects
const TIER_LARGE  = 5000;   // InstancedMesh, no custom objects
const TIER_HUGE   = 15000;  // no bloom, minimal rendering
const TIER_MASSIVE = 50000; // ngraph, absolute minimal
```

점진적 전환:
```
< 500:    custom objects + curved links + full effects + d3
500-2K:   custom objects + straight links + reduced effects + d3
2K-5K:    custom objects (LOD low) + straight links + d3, warmup 30
5K-15K:   InstancedMesh + ngraph + bloom
15K-50K:  InstancedMesh + ngraph, no bloom
50K+:     InstancedMesh + ngraph, minimal
```

### 4.3 BUG-M3: `linkColor`/`linkWidthFn`이 ref 패턴 미적용

**위치**: `GraphView3D.jsx:430-461`

`nodeThreeObject`는 `highlightSetRef.current`를 사용하여 의존성 배열에서 제거함.
그러나 `linkColor`, `linkWidthFn`, `linkOpacity`는 `highlightSet`을 **직접 의존**:

```javascript
const linkColor = useCallback(
  (link) => {
    if (!highlightSet) return base;      // ← 직접 참조
    ...
  },
  [highlightSet]                          // ← 직접 의존
);

const linkWidthFn = useCallback(
  (link) => {
    if (!highlightSet) return base;      // ← 직접 참조
    ...
  },
  [style.edgeWidthScale, highlightSet]   // ← 직접 의존
);

const linkOpacity = useMemo(
  () => {
    return highlightSet ? ... : ...;     // ← 직접 참조
  },
  [style.edgeOpacity, highlightSet, nodeCount]  // ← 직접 의존
);
```

**결과**: selectedNode 변경 시:
```
highlightSet 변경
  → linkColor 새 참조 → ForceGraph3D linkColor prop 변경 → 모든 링크 색상 재계산
  → linkWidthFn 새 참조 → ForceGraph3D linkWidth prop 변경 → 모든 링크 너비 재계산
  → linkOpacity 새 값 → ForceGraph3D linkOpacity prop 변경
  = 링크 레이어 전체 업데이트 (30K 링크면 30K × 3번 처리)
```

**수정**: nodeThreeObject와 동일한 ref 패턴 적용:
```javascript
const linkColor = useCallback(
  (link) => {
    const hl = highlightSetRef.current;
    if (!hl) return LINK_COLORS[link.type] || "#8b949e";
    // ...
  },
  []  // stable reference
);
```

### 4.4 BUG-M4: `warmupTicks = 0` → 소규모 그래프 노드 origin 시작

**위치**: `GraphView3D.jsx:491-494`

```javascript
const warmupTicks = useNgraph
  ? (nodeCount > TIER_MASSIVE ? 150 : 100)
  : (nodeCount > TIER_MED ? 60 : 0);  // ← nodeCount < 2000이면 0
```

**문제**: warmupTicks=0이면 ForceGraph3D가 노드를 배치하지 않고 즉시 렌더링을 시작한다.
d3-force의 초기 노드 위치는 `x = NaN` 또는 매우 작은 랜덤 값으로 시작하며,
cooldownTicks=200 동안 애니메이션으로 배치된다.

이 **과도기 상태에서 클릭하면 BUG-C1이 발동**한다.

```javascript
useEffect(() => {
  return () => {
    // bloom pass 해제
    if (bloomPassRef.current) {
      bloomPassRef.current.dispose?.();
      bloomPassRef.current = null;
    }

    // auto orbit 정리
    if (orbitIntervalRef.current) {
      clearInterval(orbitIntervalRef.current);
      orbitIntervalRef.current = null;
    }

    // scene extras 정리
    const extras = sceneExtrasRef.current;
    Object.values(extras).forEach((obj) => {
      if (obj) disposeObject(obj);
    });
  };
}, []);

// Three.js 객체 재귀 dispose 헬퍼
function disposeObject(obj) {
  if (obj.geometry) obj.geometry.dispose();
  if (obj.material) {
    if (Array.isArray(obj.material)) obj.material.forEach((m) => m.dispose());
    else obj.material.dispose();
  }
  if (obj.children) obj.children.forEach(disposeObject);
}
```

## 9. 요약

### 블랭크 스크린 문제의 근본 원인 체인

```
warmupTicks = 0 (소규모 그래프)
  → 노드 위치 미결정 (x=y=z=0 또는 near-zero)
  → 사용자 클릭
  → Math.hypot(0, 0, 0) = 0
  → 120 / 0 = Infinity
  → cameraPosition({ Infinity, Infinity, Infinity })
  → 화면 블랭크 (복구 불가)
```

추가 악화 요인: GraphPage.focusNode가 동일한 cameraPosition을 중복 호출하여 카메라 불안정.

### 노드 선택 시 렉의 근본 원인 체인

```
selectedNode 변경
  → highlightSet 재계산 (O(E) 인접 리스트 매번 재빌드)
  → refresh() 호출 (모든 노드 3D 객체 재생성)
  → linkColor/linkWidthFn 콜백 재생성 (prop identity 변경)
  → ForceGraph3D 내부: nodeThreeObject × N + linkColor × E + linkWidth × E
  = O(N + E) Three.js 객체 생성 + GPU 업로드
```
