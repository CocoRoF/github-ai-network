## 1. 크롤링 로직 개선 사항

### 1.1 [심각] stars 기반 필터링이 네트워크 확장을 차단함

**현재 상태:**
`fetch_user`에서 사용자의 리포지토리를 수집할 때 두 가지 필터가 적용됨:

```python
# _do_fetch_user() 내부
if self._is_ai_related(rd) or rd.get("stargazers_count", 0) > 50:
    # 이 조건을 통과해야만 수집 + fetch_contributors task 생성

# _is_ai_related() 내부
if repo_data.get("stargazers_count", 0) > 500:
    return True  # stars > 500이면 무조건 통과
```

**문제:**
- `stars > 50` / `stars > 500` 조건이 **네트워크 확장의 병목**으로 작용
- 사용자의 리포 중 이 필터를 통과하지 못한 리포→기여자→기여자의 리포... 전체 체인이 탐색 중단됨
- 결과: 네트워크에 **"사각지대"** 발생. 중요한 연결을 놓침
- stars가 낮지만 핵심 기여자가 참여하는 리포지토리가 수집 대상에서 제외됨
- 예: 기여자A가 pytorch(85k★)와 개인 프로젝트(10★)에 모두 참여 → 개인 프로젝트의 다른 기여자 체인이 영원히 미탐색

**개선 방향:**
- `_is_ai_related()`의 `stars > 500` 조건 제거
- `_do_fetch_user()`의 `stars > 50` 조건 제거
- **모든 사용자의 모든 공개 리포지토리를 수집** + 기여자 체인 확장
- 필터 자체를 제거하되, 그래프 표시 시 프론트엔드에서 min_stars 필터로 시각적 필터링

### 1.2 [심각] 서버 부담 없는 지속 가능한 크롤링 전략 부재

**현재 상태:**
- `crawler_delay = 2.0초` 고정 → 시간당 ~1800 API 호출
- GitHub Rate Limit: 인증 시 5000/hour, 미인증 시 60/hour
- Rate Limit 소진 시 reset까지 sleep (최대 ~1시간 대기)
- Worker 1개가 순차 처리 → 안전하지만 **Rate Limit 활용률이 36%에 불과**

**문제:**
- 2초 고정 딜레이는 Rate Limit 여유가 있을 때 불필요하게 느림
- Rate Limit 소진 시의 대기는 적절하지만, 소진 직전까지의 페이스를 최적화하지 않음
- 무한 확장이 목표라면 API 호출을 최대한 효율적으로 사용해야 함

**개선 방향:**
- **적응형 딜레이**: remaining이 충분하면 딜레이 줄이고, 소진 가까울 때만 늘림
- Rate Limit 예산 기반 페이싱: `remaining / (reset_time - now)` 계산으로 동적 딜레이
- 장기 크롤링 안정성: 72시간, 1주일 연속 가동에도 문제없는 구조

---

### 1.3 [중요] 기여자 수집 범위가 제한적 (15명)

**현재 상태:**
- `get_repo_contributors(per_page=15)` → 상위 15명만 수집
- 대형 프로젝트(예: tensorflow, pytorch)의 경우 수백~수천 명의 기여자 중 15명만

**영향:**
- 중요한 기여자를 놓칠 수 있음 → 네트워크 확장 경로가 좁아짐
- coworker 링크 생성이 불완전해짐
- 네트워크 밀도가 실제보다 낮아짐

**개선 방향:**
- `per_page` 증가 (30~50) 및 페이지네이션 지원 (2~3페이지)
- 더 많은 기여자 = 더 많은 확장 경로 = 더 밀도 높은 네트워크

---

### 1.4 [중요] fetch_user의 리포지토리 수집이 30개로 제한

**현재 상태:**
- `get_user_repos(per_page=30)` → 첫 페이지 30개만 조회
- 활발한 개발자는 수십~수백 개의 리포를 보유하나 일부만 수집

**개선 방향:**
- 페이지네이션 추가 (2~3페이지, 최대 90개)
- 모든 리포에서 기여자 체인 탐색 → 네트워크 확장 경로 대폭 증가

---

### 1.5 [보통] search_repos 페이지 제한

**현재 상태:**
- `search_repos`가 최대 3페이지(90개)만 수집
- GitHub Search API는 최대 1000개(34페이지) 결과를 반환 가능

**개선 방향:**
- 페이지 수 확대 또는 검색 쿼리 세분화 (stars 범위별 분할)

## 2. 그래프 생성 로직 개선 사항

### 2.1 [심각] build_graph()의 O(N) DB 쿼리 비효율

**현재 상태:**
- 매 요청마다 최대 5~6개의 DB 쿼리 실행:
  1. Repository 조회 (limit N)
  2. Author 조회 (owner_ids IN)
  3. RepoTopic 조회 (repo_ids IN)
  4. Topic 조회 (topic_ids IN)
  5. RepoContributor 조회 (repo_ids × author_ids IN)
  6. Coworker SQL 집계 (self-join)
- 캐시 TTL이 30초 → 만료 후 매번 전체 재계산

**영향:**
- 데이터 증가 시 응답 시간 선형 증가
- 특히 coworker SQL의 self-join은 O(N²) 복잡도

**개선 방향:**
- Eager loading / JOIN 기반 단일 쿼리로 통합
- coworker 관계를 별도 테이블로 사전 계산 (materialized)
- 캐시 TTL 증가 + 변경분 기반 증분 무효화
- Cursor/pagination 기반 그래프 로딩 (한번에 전체 대신 점진적 로딩)

---

### 2.2 [심각] Author 간 연결이 사실상 작동하지 않음

Author-Author 연결(coworker 링크)은 이 프로젝트의 핵심 가치이나,
현재 **3단계 구조적 결함**에 의해 대부분 생성되지 않는다.

#### 결함 1: build_graph()에서 기여자 Author가 노드에 포함되지 않음

```python
# build_graph() Step 2: owner만 수집
if r.owner_id:
    owner_ids.add(r.owner_id)
# → 기여자(contributor) Author는 노드에 추가되지 않음!
```

**결과:** `author_db_ids`가 owner만 포함 → 이후 모든 Author 관련 로직이 owner 범위로 제한됨.

#### 결함 2: coworker SQL이 owner끼리만 매칭

```python
# author_db_ids = [owner들만 포함]
coworker_sql = """
    SELECT a1.author_id, a2.author_id, COUNT(*) AS shared
    FROM repo_contributors a1
    JOIN repo_contributors a2
      ON a1.repository_id = a2.repository_id AND a1.author_id < a2.author_id
    WHERE a1.author_id IN :ids AND a2.author_id IN :ids  -- ← owner들만!
    HAVING COUNT(*) >= 2  -- ← 2개 이상 공유해야 통과
"""
```

**이중 필터 문제:**
- `WHERE ... IN :ids`: owner끼리만 검색 → 실제 기여자 쌍 대부분 탈락
- `HAVING >= 2`: 같은 repo 2개 이상 공유 필요 → owner 2명이 contributor로 2개 리포에 모두 참여하는 경우는 극히 드묾
- **결과: coworker 링크 거의 0개 생성**

#### 결함 3: get_neighbors()에서 author 클릭 시 coworker 탐색 없음

```python
elif node_type == "author":
    # ✅ 소유 리포 표시 (owns)
    # ✅ 기여 리포 표시 (contributes)
    # ❌ coworker 탐색 없음 — 같은 repo의 다른 contributor를 찾지 않음!
    return {...}
```

**결과:** 사용자가 Author 노드를 클릭해도 해당 개발자와 함께 일한 다른 개발자를 전혀 볼 수 없음.

#### 구체적 시나리오

```
실제 데이터: pytorch/pytorch의 기여자 = A, B, C, D, E...
             huggingface/transformers의 기여자 = A, F, G, H...
             → A는 양쪽 모두에 기여 → A-B, A-C, A-F... 모두 coworker

현재 동작:
  1. build_graph()가 repo 노드 생성
  2. owner_ids = {pytorch_owner, transformers_owner} (2명만)
  3. A, B, C, D, E, F, G, H → 전부 노드 미포함
  4. author_db_ids = [pytorch_owner, transformers_owner]
  5. coworker SQL: 이 2명이 함께 contribute한 repo 검색 → 대부분 0개
  6. 결과: contributes 링크 0개, coworker 링크 0개
```

**결론:** 크롤러가 열심히 수집한 RepoContributor 데이터가 그래프에서 전혀 활용되지 않고 있음.

#### 개선 방향 (3가지 동시 수정 필요)

| 대상 | 현재 | 개선 |
|------|------|------|
| **build_graph() Author 노드** | owner만 | owner + 기여자 (contributions ≥ N 기준) |
| **build_graph() coworker SQL** | owner끼리, HAVING ≥ 2 | 전체 author, HAVING ≥ 1 |
| **get_neighbors() author** | owns + contributes만 | + coworker 탐색 추가 |

### 2.3 [보통] 노드 크기 산정의 다양성 부족

**현재 상태:**
- Repository: `sqrt(stars)` → 100★ = 10, 10000★ = 100, 100000★ = 316
- 극단적으로 큰 리포(tensorflow: 190k★)와 중간 리포(1k★)의 시각적 차이가 큼

**개선 방향:**
- log 스케일 적용: `log10(stars + 1) × 10`
- 타입별 정규화 (각 타입 내에서 min-max 또는 percentile)
- 사용자 조절 가능한 크기 파라미터

---

### 2.4 [보통] Language 데이터 미활용

**현재 상태:**
- Repository.language 필드를 저장하고 필터로는 사용 가능
- 하지만 Language 자체가 노드로 존재하지 않음

**개선 방향:**
- Language 노드 추가 (선택적)
- 같은 언어의 리포지토리 클러스터 시각화
- Language 필터와 연동

### 2.5 [보통] Organization 데이터 미활용

**현재 상태:**
- Author.company 필드를 저장하지만 그래프에서 미사용
- Author.user_type에 "Organization" 구분 있음

**개선 방향:**
- Organization 노드 만들어 같은 조직의 개발자 클러스터 시각화
- company 필드 기반 매칭 (정규화 필요: "@google" → "Google")

## 3. 인프라/아키텍처 개선 사항

### 3.1 [심각] SQLite의 동시성 한계

**현재 상태:**
- SQLite를 aiosqlite로 비동기 사용
- 쓰기 연산 시 전체 DB lock → Worker 쓰기와 API 읽기가 상호 블로킹 가능

**영향:**
- 크롤링 중 그래프 API 응답 지연
- 대규모 데이터(10만+ 레코드)에서 성능 저하

**개선 방향:**
- PostgreSQL로 마이그레이션 (프로덕션 환경)
- WAL(Write-Ahead Logging) 모드 활성화 (SQLite 유지 시)
- 읽기 전용 복제본 사용

---

### 3.2 [중요] 캐시 전략의 단순함

**현재 상태:**
- 단일 dict 기반 인메모리 캐시
- TTL 30초 고정
- task 완료 시 전체 캐시 무효화 (모든 키 삭제)

**문제:**
- 크롤링 active 상태에서는 캐시가 거의 무의미 (매 task 완료 시 invalidate)
- 세션별 캐시 분리 없음

**개선 방향:**
- 세션별 캐시 무효화 (다른 세션 데이터는 유지)
- Redis 기반 캐시 (스케일 아웃 시)
- 변경된 노드만 증분 갱신

---

### 3.3 [보통] 에러 복구 전략

**현재 상태:**
- task 처리 실패 시 `status="error"` 마킹 후 다음 task로 이동
- 실패한 task는 재시도되지 않음 (영구 실패)
- Worker 전체 crash 시 10초 후 재시도

**개선 방향:**
- task 재시도 메커니즘 (retry_count, max_retries)
- 실패 task를 일정 시간 후 pending으로 리셋하는 스케줄러
- 특정 오류 타입에 따른 차등 재시도 (네트워크 오류 vs. 404)

---

### 3.4 [보통] API 호출 효율

**현재 상태:**
- `_save_repo()` 내부에서 owner마다 `_save_author()` 호출 → 개별 DB 조회
- `_save_repo()` 호출 시 매번 topics 개별 조회 및 삽입
- `_do_fetch_contributors()` 내에서 contributor마다 개별 DB upsert

**개선 방향:**
- 배치 upsert (bulk insert with on_conflict)
- 사전 캐시: 현재 세션에서 이미 조회된 author_id / topic_name 캐시
- 불필요한 SELECT 제거 (INSERT OR IGNORE 패턴)

## 4. 기능 확장 개선 사항

### 4.1 [중요] 크롤링 깊이/범위의 가시성 부족

**현재 상태:**
- depth를 기록하지만 UI에서 depth별 분포를 보여주지 않음
- 크롤링이 어디까지 확장되었는지 시각적으로 파악 불가

**개선 방향:**
- task depth 히스토그램 API
- 그래프에서 depth별 색상 구분
- 세션 상세에서 depth별 노드 수 표시

---

### 4.2 [보통] 크롤링 전략의 유연성 부족

**현재 상태:**
- 모든 세션이 동일한 확장 로직 사용
- 세션별로 다른 전략(bfs-only, dfs 등) 선택 불가

**개선 방향:**
- 세션 생성 시 전략 선택:
  - `wide`: 검색 결과만 수집 (fetch_contributors 생략)
  - `full` (기본): 모든 리포 + 기여자 체인 무한 확장
  - `contributors-only`: 기여자 관계만 수집 (user 리포 탐색 생략)

---

### 4.3 [보통] 데이터 신선도 관리 없음

**현재 상태:**
- `crawled_at` 필드가 있지만 데이터 갱신은 재크롤링 시에만 발생
- 오래된 데이터(stars 수 변화, 삭제된 리포 등) 감지 불가

**개선 방향:**
- 정기적 re-crawl 세션 (crawled_at이 N일 이상된 엔티티만 갱신)
- 삭제/아카이브된 리포지토리 마킹
- 변화 추적 (delta) 기능

---

### 4.4 [낮음] 그래프 내보내기/공유 기능 없음

**개선 방향:**
- GraphML, GEXF, JSON 형식 내보내기 API
- 특정 세션의 그래프 스냅샷 저장
- 공유 가능한 URL 생성

---

### 4.5 [낮음] 다중 Worker 확장성

**현재 상태:**
- 단일 Worker → 단일 서버에서 초당 처리 속도 제한적 (delay 2초)
- 시간당 최대 ~1800 API 호출 (5000 한도 대비 36%)

**개선 방향:**
- 다중 Worker (동시에 N개 task 처리)
- Rate Limit을 Worker 간 공유하는 토큰 버킷 패턴
- 여러 GitHub 토큰 풀 사용

---

## 5. 개선 사항 우선순위 매트릭스

> 우선순위 기준: "무한 확장을 서버 부담 없이 지속할 수 있는가?"

| # | 개선 사항 | 심각도 | 구현 난이도 | 추천 순서 |
|---|----------|--------|-----------|----------|

## 6. 요약

현재 시스템의 무한 확장 설계는 프로젝트의 핵심 목표에 부합하는 강점이다.
그러나 **stars 기반 필터링이 확장 경로를 인위적으로 차단**하여 네트워크에 사각지대가 발생하고 있다.
가장 시급한 개선은 필터 제거(1순위)와 Rate Limit 효율화(2순위)로,
이 두 가지만으로도 네트워크 확장 속도와 밀도가 크게 향상될 수 있다.
그래프 측에서는 기여자 Author 노드 누락(3순위)이 contributes/coworker 링크를 살려내는 핵심 개선이다.
