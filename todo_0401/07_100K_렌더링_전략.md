# 100K+ 노드 렌더링 전략: 근본적 아키텍처 전환

## 1. 핵심 문제: react-force-graph-3d의 구조적 한계

`three-forcegraph` 소스코드 확인 결과, **노드 하나당 개별 THREE.Mesh 생성** (라인 1143: `obj = new three$1.Mesh()`).
InstancedMesh는 어디에도 사용되지 않음.

| 노드 수 | Draw Calls | 결과 |
|---------|-----------|------|
| 5,000 | 5,000+ | 눈에 띄는 프레임 드롭 |
| 10,000 | 10,000+ | 거의 사용 불가 |
| 50,000 | 50,000+ | WebGL 크래시 가능 |
| 100,000 | 100,000+ | 불가능 |

**현재 GraphView3D.jsx의 티어 시스템**은 기능을 점진적으로 비활성화하지만,
**근본 병목(노드당 1 draw call)은 해결 불가**.

---

## 2. 새로운 아키텍처: 2 Draw Calls로 전체 그래프 렌더링

### 핵심 원칙
```
기존: 100,000 Mesh + 200,000 Line = 300,000 draw calls
신규: 1 InstancedMesh + 1 LineSegments = 2 draw calls
```

### 2.1 아키텍처 다이어그램

```
[Web Worker]                              [Main Thread - Three.js Scene]
  d3-force-3d simulation                    ┌─────────────────────────┐
  Barnes-Hut O(n log n)                     │  1x InstancedMesh       │
  Posts Float32Array @ 30Hz  ──────────►    │    (모든 노드)          │
  via transferable objects                  │  1x LineSegments        │
                                            │    (모든 엣지)          │
                                            │  Frustum Culling        │
                                            │  GPU Picking            │
                                            └─────────────────────────┘
```

### 2.2 노드 렌더링: InstancedMesh + Custom Shader

**InstancedMesh**는 동일한 geometry를 수만 번 렌더링할 때,
GPU가 **단 1번의 draw call**로 모든 인스턴스를 처리.

Per-instance attributes:
- `instanceOffset` (vec3): 노드 위치
- `instanceColor` (vec3): 노드 타입별 색상
- `instanceScale` (float): 노드 크기
- `instanceHighlight` (float): 선택/하이라이트 상태

선택/하이라이트 변경 시:
- 오브젝트 재생성 없음
- `instanceHighlight` 배열만 업데이트 → `needsUpdate = true`
- GPU가 다음 프레임에서 자동 반영

### 2.3 엣지 렌더링: LineSegments

하나의 `BufferGeometry`에 모든 엣지의 시작/끝 좌표를 저장.
노드 위치 업데이트 시 엣지 좌표도 동기화.

### 2.4 Force Layout: Web Worker

메인 스레드에서 force simulation 제거.
Worker에서 계산 → `Float32Array`를 **transferable object**로 전송 (zero-copy).

```
Worker tick → Float32Array(nodeCount * 3) → postMessage([buffer])
Main thread → offsetAttr.array.set(positions) → needsUpdate = true
```

### 2.5 노드 선택: GPU Picking

100K 노드에서 Three.js raycasting은 O(N) → 수백ms 소요.

GPU Picking:
1. 오프스크린 렌더 타겟에 각 노드를 고유 색상(인덱스 인코딩)으로 렌더
2. 마우스 좌표의 픽셀 색상 읽기 → 노드 인덱스 역산
3. O(1) — 노드 수와 무관

### 2.6 Frustum Culling

카메라 시점 밖의 노드는 렌더링하지 않음.
InstancedMesh의 `count`를 가시 노드 수로 조정.

줌인 시 전체 100K 중 가시 영역의 5K-10K만 렌더 → GPU 부하 70-80% 감소.

## 3. 마이그레이션 전략

### 두 컴포넌트 병행

| 노드 수 | 컴포넌트 | 이유 |
|---------|---------|------|
| < 5,000 | GraphView3D.jsx (기존) | react-force-graph-3d가 잘 작동, 풍부한 시각 효과 |
| >= 5,000 | GraphView3DLarge.jsx (신규) | 커스텀 Three.js, 2 draw calls |

GraphPage.jsx에서 노드 수에 따라 자동 전환.

---

## 4. 구현 순서

| 단계 | 작업 | 세부 |
|------|------|------|
| 1 | Web Worker 레이아웃 | d3-force-3d를 Worker에서 실행, position transfer |
| 2 | InstancedMesh 노드 렌더 | Custom ShaderMaterial, per-instance attributes |
| 3 | LineSegments 엣지 렌더 | BufferGeometry, 노드 위치와 동기화 |
| 4 | 카메라/컨트롤 | OrbitControls, zoomToFit |
| 5 | GPU Picking | 오프스크린 렌더, 클릭→노드 ID |
| 6 | 선택/하이라이트 | instanceHighlight 업데이트 |
| 7 | 시각 효과 | Bloom, Star field, Fog (선택적) |
| 8 | GraphPage 통합 | 노드 수 기반 컴포넌트 전환 |

## 5. 예상 성능

| 지표 | 현재 (react-force-graph-3d) | 신규 (Custom) |
|------|--------------------------|--------------|
| Draw calls (100K 노드) | ~100,000 | 2 |
| 메인 스레드 차단 | Force tick마다 | 없음 (Worker) |
| 노드 선택 지연 | O(N) raycasting | O(1) GPU pick |
| 하이라이트 갱신 | 오브젝트 재생성 | attribute 업데이트 |
| 메모리 | N × Mesh 오버헤드 | 1 InstancedMesh |
| 목표 FPS | 5-10 @ 30K | 60 @ 100K+ |
